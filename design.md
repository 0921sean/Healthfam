## Problem Statement

헬스가 하기 싫지만 해야 한다는 걸 아는 사람들이 있다. 혼자서는 작심삼일로 끝난다. 근데 친구들과 함께하면 다르다: 돈을 내기 싫어서라도 헬스를 간다. 실패해도 괜찮다: 그 주에 못 간 사람이 회식을 사면 된다. 어차피 친구들을 만나게 된다. 이 앱은 그 시스템을 수동(카카오톡 그룹채팅 + 카카오뱅크 수동 정산)에서 자동으로 전환한다.

## Demand Evidence

- 창업자가 3년째 친구 그룹과 직접 운영 중. 실제 돈이 오가고, 이탈 없음.
- 매 주 헬스 횟수 카운팅, 벌금 계산, 정산을 수동으로 운영 중. 이 불편함이 앱의 존재 이유.
- 주기적 회식이 자연스럽게 발생하고 있으며, 참여 못 한 멤버에게도 1/n을 보내는 등 자체적인 규칙 진화가 일어남.

## Status Quo

- 인증: 카카오톡 그룹채팅에 사진 올리기
- 정산: 창업자(방장)가 매주 수동으로 횟수 카운팅 → 벌금 계산 → 카카오뱅크 이체 요청
- 회식 정산: 불참자에게 1/n 금액 별도 이체

불편한 것:
1. 방장이 매주 손으로 세야 함
2. 얼마인지 카카오톡으로 일일이 알려줘야 함
3. 예외 처리(군대, 여행, 부상)를 사람이 직접 판단하고 기록해야 함

## Target User

초기: 창업자의 친구 그룹 (20대 한국 남성, 헬스를 해야 한다는 걸 알지만 동기가 부족한 사람들)
확장: 비슷한 구조로 운영하고 싶은 다른 친구 그룹들

"헬스가 재미없지만 해야 하는 사람들에게 재미 요소를 가미해주는 서비스"

## Narrowest Wedge

모임 만들기 + 사진 인증 + 벌금 자동 계산. 이 세 개만 있으면 카카오톡 수동 운영을 대체할 수 있다.

## Premises

1. **핵심 동기는 사회적 압박이다** — "내가 회식에서 사는 사람"이 되기 싫다는 게 진짜 동기. 돈은 그걸 가시화하는 도구.
2. **느슨한 사진 인증은 피처다** — 신뢰 기반 친구 그룹에서는 피드가 보이는 것 자체가 충분한 accountability. "사진 못 찍었다"는 것도 다 봐준다.
3. **MVP의 경쟁자는 카카오톡 + 수동 정산** — 다른 피트니스 앱이 아니라, 지금의 수동 운영 방식이 대체 대상.

## Cross-Model Perspective

Codex 두 번째 의견 (gpt-5.3-codex, read-only):

"This is a behavior product, not a fitness product: it operationalizes friend-group shame into a weekly ritual with stakes. Penalties are just the visible scoreboard that turns 'I should go' into 'I can't be the one buying dinner.' You already validated the core loop for 3 years in the wild, which is stronger than most MVP hypotheses."

핵심 지적: "build priority is reducing coordinator overhead, not inventing motivation mechanics." → 동의. 동기 메커니즘은 이미 3년째 검증됨. 앱의 임무는 방장의 수고를 없애는 것.

스트레스 테스트: 사진 검증의 느슨함은 친밀한 친구 그룹에서는 통하지만 모르는 사람들 사이에서는 깨질 수 있음. MVP 범위(친구 그룹)에서는 문제없지만, 스케일 시 다시 평가 필요.

## Approaches Considered

### Approach A: PWA MVP
Next.js + Supabase로 모바일 웹앱. 빠르지만 iOS 푸시 알림 불안정, 카메라 UX 제한적.

### Approach B: React Native / Expo (선택)
Expo로 iOS/Android 동시 개발. 진짜 푸시 알림, 네이티브 카메라 접근.

### Approach C: 카카오 미니앱
카카오톡 내부에서 동작. 마찰 최소화지만 플랫폼 의존도 높고 진입장벽 높음.

## Recommended Approach

**React Native + Expo + Supabase**

이유: Savage/Awkward 알림이 핵심 차별화이므로 iOS 푸시 알림이 안정적이어야 함. 사진 촬영도 네이티브 카메라 접근이 필요. PWA는 이 두 가지에서 제한적.

### 기술 스택

- **Frontend:** React Native + Expo (TypeScript)
- **Backend:** Supabase (PostgreSQL, Auth, Storage, Realtime)
- **Push 알림:** Expo Notifications + FCM/APNs
- **인증:** 휴대폰 번호 + SMS OTP (MVP). 카카오 OAuth는 v1.1 이후 추가 (카카오 OAuth는 scope/심사 등 진입장벽 있음)
- **배포:** Expo EAS Build → TestFlight (iOS), Internal Track (Android)

### 핵심 화면 및 기능

**1. 모임 만들기**
- 모임 이름, 주간 목표 횟수, 회당 벌금액
- 알림 모드: 사나운 모드 / 어색한 사이 모드
- 초대 링크 생성 → 카카오톡으로 공유

**2. 인증 피드**
- 사진 업로드 (카메라 직접 or 갤러리)
- 피드: 이번 주 멤버들의 인증 사진 모아보기
- "오늘 몇 명 갔나" 한눈에 확인

**3. 벌금 현황**
- 이번 주 현황: 멤버별 인증 횟수 / 목표 횟수
- 자동 계산: (목표 - 실제) × 벌금액 = 이번 주 벌금
- 누적 벌금 (회식 적립금)

**4. 예외 처리 (방장 권한)**
- 흐름: 멤버가 앱 내에서 예외 신청(사유 선택: 군대/여행/부상/기타) → 방장이 앱에서 승인/거절
- 승인 시: 해당 주 목표 횟수를 방장이 직접 숫자로 조정 (0~원래 목표)
- 타이밍: 주 중 언제든 신청 가능. 마감(월요일 00:00 KST) 전까지 처리하면 그 주 정산에 반영.
- 복잡한 자동화 없음. 방장 판단 = 최종.

**5. 주간 마감**
- 매주 월요일 00:00 KST 자동 마감 (Supabase Edge Function cron)
- 이전 주 체크인 잠금 (새 사진 업로드 불가)
- 벌금 확정: 각 멤버의 (목표 - 실제 체크인 수) × 벌금액 → penalties 테이블 기록
- 회식 적립금 = 모든 멤버 주간 벌금 합산 (누적)
- 주간 마감 요약 알림 발송: "[이름]님 이번 주 [N]번, 벌금 [N]원. 현재 회식 적립금 [N]원"
- 신규 주 자동 시작 (새 checkins 슬롯 오픈)
- 중간에 합류한 멤버: 합류 주부터 목표 적용, 이전 주 소급 없음

**6. 알림 (차별화 포인트)**

알림 모드는 그룹 생성 시 방장이 선택. MVP에서 두 모드 모두 출시 (이게 핵심 차별화).

사나운 모드:
- "야 헬스 안 가냐? 이번 회식도 네가 사려고?"
- "오늘까지 [N]번 더 가야 하는데. 벌금 [N]원 쌓이는 중"

어색한 사이 모드:
- "헬스... 안 가...?"
- "이번 주 목표까지 [N]번 남았어요..."

알림 트리거: 수요일 저녁 (주 중반 리마인드), 토요일 오전 (주말 마지막 기회), 월요일 00:05 KST (정산 완료 통보)

### 데이터 모델 (간략)

```
groups: id, name, weekly_target, penalty_per_miss, mode, created_by
members: group_id, user_id, role (admin/member)
checkins: group_id, user_id, photo_url, checked_at, week_number
exemptions: group_id, user_id, week_number, reason, reduced_target
penalties: group_id, user_id, week_number, amount (계산값)
```

### 실시간 처리 범위

Supabase Realtime은 텍스트 메타데이터만 (체크인 수, 벌금 현황). 사진은 on-demand 로드 또는 10초 폴링. 배터리 소모/RLS 한계 방지.

### 의심 사진 처리

AI 검증 없음. 단, 마감 후 방장은 체크인 피드에서 특정 사진에 "이의 제기" 플래그 가능. 플래그된 항목은 벌금 확정 전 방장이 수동으로 결정. MVP에서 복잡한 자동화는 없음.

### 스킵할 것 (MVP)

- 실제 결제 연동 (KakaoPay API 등) — 숫자만 보여주고 실제 이체는 카카오뱅크에서 직접
- 사진 AI 검증
- 웨어러블 연동
- 복잡한 예외 자동화
- 디자인 시스템 완성도
- 카카오 OAuth (MVP는 휴대폰 번호 인증으로 시작)

## Success Criteria

- 창업자의 기존 친구 그룹이 카카오톡 수동 운영을 앱으로 전환 (1개 그룹)
- 2주 이상 앱으로만 운영 가능
- 방장이 매주 수동으로 계산하는 시간 = 0

## Privacy & Data Retention

- 사진은 Supabase Storage에 비공개 저장 (그룹 멤버만 접근 가능)
- 사진은 12주 후 자동 삭제 (회식 주기 고려)
- 사용자 데이터(휴대폰 번호, 사진)는 외부에 판매/공유 없음
- 그룹 탈퇴 시 해당 멤버의 사진 및 체크인 데이터 삭제 가능 (방장 요청)

## Dependencies

- Kakao 개발자 계정 (카카오톡 공유용 — 링크 공유만, OAuth는 나중에)
- Supabase 휴대폰 OTP 설정 (Twilio 또는 직접 SMS provider)
- Supabase 프로젝트 셋업
- Apple Developer 계정 (TestFlight)
- Google Play Developer 계정

## The Assignment

이번 주 안에 본인 친구 그룹에 Expo Go 링크 혹은 TestFlight 초대장을 보내고 직접 써보게 해라. 수동 카카오톡 운영을 앱으로 대체하는 첫 번째 그룹을 만드는 것. 코드 먼저가 아니라, 지금 운영 중인 그 그룹이 첫 번째 사용자다.

## What I noticed about how you think

- "돈을 내기 싫어서 계속 헬스를 가게 되는 느낌" — 동기부여가 아니라 손실 회피를 메커니즘으로 쓴다는 걸 직관적으로 알고 있었음. 대부분의 사람은 이걸 의식화하지 못하면서 앱을 만든다.
- 사나운 모드 / 어색한 사이 모드 — 알림 카피를 관계의 온도에 맞게 달리 설계한다는 발상. UX 디테일에 대한 감각이 있다.
- 군대, 헬스장 폐업, 여행, 부상 — 3년 운영하면서 만들어진 예외 규칙들을 구체적으로 기억하고 있음. 이게 진짜 product spec이다.
- "사진이 헬스 사진이 맞는지는 딱히 검사 안 할 거임" — 기술적 엄밀함보다 사회적 신뢰를 선택한 것. 이 제품이 어떤 관계 위에서 작동하는지 알고 있다는 뜻.
